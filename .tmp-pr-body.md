Adds a new `tailwindCSS.experimental.v4Root` setting so v4 entrypoint detection can be overridden from VS Code settings while keeping the current behavior as the default.

Right now we hard-code v4 entrypoint detection around `@import "tailwindcss"` / `@import "tailwindcss/..."` in a couple of places. That works for the common case, but it makes automatic project discovery too rigid for users who intentionally use a different entrypoint marker.

This change moves that logic behind a shared matcher and threads workspace settings through the relevant code paths so project discovery, restart detection, and v4 design system initialization all agree on what counts as a v4 root.

Example:

```json
{
  "tailwindCSS.experimental.v4Root": [
    "@reference\\s*['\"]tailwindcss['\"]"
  ]
}
```

Also included:

- docs + VS Code setting metadata
- tests for the default matcher, a custom matcher case, and non-matching imports like `@import "my-self-tailwindcss"`

Tested with:

- `pnpm --filter @tailwindcss/language-server exec vitest run src/version-guesser.test.ts`
- `pnpm --filter @tailwindcss/language-server exec vitest run src/version-guesser.test.ts tests/env/v4.test.js -t "custom v4 root detection"`

I also ran `pnpm --filter @tailwindcss/language-server exec tsc --noEmit`, but that currently reports existing repo/environment TypeScript issues unrelated to this change.
